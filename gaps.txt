The Shellock project claims to be an "adaptive" and "scrutable" LAM but several
    loops are incomplete. Please identify and implement the following:

    1. CLOSED FEEDBACK LOOPS — After a `shellock fix` succeeds, record the
       (error_fingerprint → fix) pair in a per-user knowledge base
       (~/.shellock/learned_fixes.json). On future `shellock fix` calls, check this
       before calling the LLM. Show the user "[LEARNED] This fix worked before" when
       a learned fix is applied.

    2. RICHER PREFERENCE CONTEXT — When calling the LLM for spec generation, the
       current user_preferences dict only passes tool frequency counts. Extend it to
       also pass: (a) the last 3 approved specs as examples, (b) any packages the
       user has manually added/removed via edits. The LLM should use these as
       few-shot examples.

    3. CONTRASTIVE SCRUTABILITY — When `[ADAPT:sys]` announces a choice (e.g.
       "selected python module"), also log what was considered and rejected, e.g.
       "[ADAPT:sys] Selected python (rejected: node — no package.json found)".
       Apply this to module selection, LLM tier selection, and package suggestions.

    4. SPEC DIFF ON RE-INIT — If a `.shellock/spec.json` already exists in the
       project and the user runs `shellock init` again, show a diff of what changed
       (old spec vs new spec) in the approval screen rather than just the new plan.
       Make the user explicitly confirm they want to replace it.

    5. PROACTIVE ADAPTATION ANNOUNCEMENT — Currently adaptations are silent unless
       the user reads the [ADAPT] lines. Add a `shellock why` command that explains
       in plain English why the last action was set up the way it was: which
       preferences influenced it, which context signals were detected, and what would
       have happened under different conditions (e.g. "If you hadn't had pyenv, we
       would have used the system Python").

    Keep all changes consistent with the existing architecture: Pydantic schemas for
    any new data structures, human approval gates before any new automated behavior,
    and audit trail entries for any new actions.

    Biggest Gaps (High Impact)

  1. Cross-project error learning (learned.json is empty)

  Errors and their working fixes are tracked per-project but never shared globally. If pip 
  install fastapi fixes "No module named fastapi" in one project, that knowledge should    
  transfer automatically to all projects.

  2. No time-decay on preferences

  A tool used 3 years ago is weighted the same as one used yesterday. Old choices never    
  fade, so suggestions become stale over time.

  3. Suggestions aren't context-aware

  All suggestions come from a flat "tools" bucket — Shellock doesn't notice that you always
   use black for web projects but ruff for ML projects. Project type isn't tracked or used.

  4. No feedback loop on suggestions

  When a suggestion is auto-added or rejected, nothing is learned about why.
  rejected_suggestions is just a blocklist — it doesn't distinguish "wrong tool" from      
  "wrong timing."

  5. Rollback doesn't affect preference weights

  If a user rolls back a shellock init that auto-included a suggested tool, the preference 
  counter isn't decremented. Bad suggestions keep getting made.

  6. Hardware detection doesn't drive recommendations

  GPU/CUDA is detected and displayed but never used to adapt suggestions (e.g., auto-adding
   torch[cuda] when CUDA is available, or recommending GPU-accelerated alternatives).
